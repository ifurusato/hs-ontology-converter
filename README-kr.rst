Understanding the Ontology YAML Format
======================================

The YAML files generated by this project use a knowledge representation
format developed for the Kraken ontology system. This document describes
enough of the format for a reader to understand and consume the generated
files without access to the Kraken system itself.

The Kraken ontology is a private system and is not included in or
distributed with this repository. The ``kr:`` namespace terms referenced
throughout the generated files are symbolic identifiers grounded in that
system. Their semantics are described here in plain language.


File Structure
--------------

Every ontology YAML file begins with an ``ontology:`` header block that
declares the file's identity and its namespace context map::

    ontology:
        version:     1.0
        id:          hs
        title:       Harmonized System Ontology
        description: "..."
        author:      "..."
        created:     2026-01-01
        contexts:
            hs:   $HS_TERM_NAMESPACE_URI
            kr:   $KRAKEN_TERM_NAMESPACE_URI

The ``contexts:`` block maps namespace prefixes to URI bases. The
``$VAR_NAME`` values are resolved by the Kraken runtime; for consumers
of the raw YAML, the prefixes can be read as stable symbolic namespaces.

Following the header, each term is defined as a top-level YAML key of
the form ``namespace:name`` (e.g. ``hs:section``, ``nztar:rate``).


Term Definition
---------------

Each term is a top-level YAML key followed by its attributes::

    hs:section.s1:
        name:        section.s1
        display:     Section I
        description: "Live animals; animal products"
        identity:    "urn:neocortext:term:hs:section.s1"
        properties:
            kr:instanceOf:  hs:section
            hs:isSectionOf: hs:tradedGood

The standard attributes are:

.. list-table::
   :header-rows: 1
   :widths: 20 10 70

   * - Attribute
     - Required
     - Description
   * - ``name``
     - Yes
     - The local name without namespace prefix
   * - ``display``
     - Yes
     - Human-readable label
   * - ``description``
     - No
     - Full definition of the term
   * - ``comment``
     - No
     - Additional explanatory notes
   * - ``identity``
     - Yes
     - Canonical URN uniquely identifying this term
   * - ``properties``
     - No
     - Relations from this term to other terms


Properties
----------

The ``properties:`` block declares typed relations from the term to other
terms or to scalar values. A property value is either a single term
reference, a list of term references, or a scalar::

    # Single reference
    properties:
        kr:kindOf: kr:concept

    # Multiple references
    properties:
        kr:kindOf:
        - kr:thing
        - kr:truthFunction

    # Scalar values
    properties:
        kr:validFrom:  "2012-01-01"
        kr:validTo:    "9999-12-31"
        isPredicate:   true
        isTransitive:  false

Term references are written as ``namespace:local`` and resolve to the
term defined elsewhere in the same file or in an imported ontology file.


Node Key Naming
---------------

All term keys are XML Name compliant — the local part after the namespace
prefix does not begin with a digit. Numeric HS code segments are prefixed
with a short alphabetic discriminator:

.. list-table::
   :header-rows: 1
   :widths: 15 30 55

   * - Prefix
     - Node type
     - Example
   * - ``s``
     - Section
     - ``hs:section.s1``
   * - ``c``
     - Chapter
     - ``hs:chapter.c01``
   * - ``h``
     - Heading
     - ``hs:heading.h0101``
   * - ``sh``
     - Subheading
     - ``hs:subheading.sh010190``
   * - ``ti``
     - Tariff Item
     - ``hs:tariffItem.ti01019000``
   * - ``sc``
     - Statistical Code
     - ``hs:statisticalCode.sc0101900090D.v20020101``
   * - ``r``
     - Rate
     - ``nztar:rate.r0101100011.NML.v20020101``
   * - ``l``
     - Levy
     - ``nztar:levy.l0102210010.AS.v20120101``
   * - ``lf``
     - Levy Formula
     - ``nztar:levyFormula.lf1``

Versioned nodes (statistical codes, rates, levies) carry a ``.v{date}``
suffix indicating the start date of that version's validity period, in
the form ``vYYYYMMDD``.


Core ``kr:`` Terms
------------------

The following ``kr:`` namespace terms appear throughout the generated
files. They are defined in the Kraken upper ontology; their semantics
are summarised here for reference.

**Classification:**

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``kr:kindOf``
     - The subject term is a subtype or specialisation of the object
       term. This is an intensional (definition-based) relation, not
       extensional (membership-based). Transitive.
   * - ``kr:instanceOf``
     - The subject term is an instance of the object term as a class.
   * - ``kr:concept``
     - An abstract, organised idea. The base type for all HS
       classification nodes in this project.
   * - ``kr:thing``
     - The root of all terms in the upper ontology.
   * - ``kr:predicate``
     - A relation between a subject and an object.
   * - ``kr:predicate.kindOf``
     - A taxonomic subsumption relation — a specialisation of
       ``kr:predicate`` used for hierarchy relations.

**Temporal:**

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``kr:validFrom``
     - ISO 8601 date from which this term version is valid.
   * - ``kr:validTo``
     - ISO 8601 date until which this term version is valid.
       The sentinel value ``9999-12-31`` indicates currently active
       with no known end date.

**Predicate flags:**

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``isPredicate``
     - Boolean. This term is a relation type (predicate).
   * - ``isTransitive``
     - Boolean. This predicate is transitive: if A→B and B→C
       then A→C.


``hs:`` Terms
-------------

These terms are defined in ``hs.yaml`` and used throughout
``tariff_details.yaml``.

**Node types** — all ``kr:kindOf kr:concept``:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``hs:tradedGood``
     - Root node. Any thing that may appear on a customs declaration,
       manifest, or waybill. All HS sections are subcategories of this.
   * - ``hs:nomenclature``
     - A specific edition of the WCO HS Nomenclature.
   * - ``hs:section``
     - A Section of the HS Nomenclature (21 sections, identified by
       Roman numeral).
   * - ``hs:chapter``
     - A Chapter (2-digit grouping within a Section).
   * - ``hs:heading``
     - A Heading (4-digit code within a Chapter).
   * - ``hs:subheading``
     - A Subheading (6-digit code within a Heading). This is the limit
       of the internationally standardised WCO HS code.
   * - ``hs:tariffItem``
     - An 8-digit NZ Customs Tariff Item extending the international
       subheading.
   * - ``hs:statisticalCode``
     - A 10-digit NZ statistical code; the leaf node of the hierarchy.

**Hierarchy relations** — all ``kr:kindOf kr:predicate.kindOf``:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``hs:isSectionOf``
     - Subject Section is a subcategory of ``hs:tradedGood``.
   * - ``hs:isChapterOf``
     - Subject Chapter is a subcategory of the object Section.
   * - ``hs:isHeadingOf``
     - Subject Heading is a subcategory of the object Chapter.
   * - ``hs:isSubheadingOf``
     - Subject Subheading is a subcategory of the object Heading.
   * - ``hs:isTariffItemOf``
     - Subject Tariff Item is a subcategory of the object Subheading.
   * - ``hs:isStatisticalCodeOf``
     - Subject Statistical Code is a subcategory of the object
       Tariff Item.

**Temporal relations** — ``kr:kindOf kr:predicate``:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``hs:isAmendedBy``
     - The subject node has been administratively amended by the object
       node. Both share the same HS code key; only the description or
       metadata has changed. Present only in history mode output.
   * - ``hs:isReplacedBy``
     - The subject node has been abolished and replaced by the object
       node, typically at a WCO revision boundary.

**Cross-reference relation** — ``kr:kindOf kr:predicate``:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``hs:hasAlternate``
     - The subject node has an alternate classification in the object
       node, as indicated by the NZ Customs alternate tariff item field.


``nztar:`` Terms
----------------

These terms are defined in ``hs.yaml`` and used throughout
``tariff_rates.yaml`` and ``tariff_levies.yaml``.

**Node types** — all ``kr:kindOf kr:concept``:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``nztar:rate``
     - A duty rate applicable to an HS tariff item within a validity
       period under the NZ Working Tariff.
   * - ``nztar:levy``
     - A levy applicable to an HS tariff item within a validity period.
   * - ``nztar:levyFormula``
     - A formula used to calculate a levy amount.

**Relations** — all ``kr:kindOf kr:predicate``:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Term
     - Meaning
   * - ``nztar:appliesToCode``
     - The subject rate or levy applies to the object HS tariff item.
   * - ``nztar:usesFormula``
     - The subject rate or levy is calculated using the object levy
       formula.
   * - ``nztar:hasRate``
     - The subject HS tariff node has the object rate applicable within
       a validity period.
   * - ``nztar:hasLevy``
     - The subject HS tariff node has the object levy applicable within
       a validity period.


Temporal Queries
----------------

Every versioned node carries ``kr:validFrom`` and ``kr:validTo``
properties. To reconstruct the HS taxonomy as it stood on a given date
*T*, select all nodes where ``validFrom ≤ T ≤ validTo`` and traverse
the hierarchy relations only through nodes also valid at *T*.

The sentinel value ``9999-12-31`` for ``kr:validTo`` indicates a
currently active node with no known expiry. Historical nodes (those with
a ``kr:validTo`` in the past) are only present in the
``ontology_history/`` output generated with the ``--include-history``
flag.
